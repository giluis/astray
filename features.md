## Basics


### The Parsable<T> trait

At the heart of Astray lies the `Parsable<T>` trait. Check its definition [here](https://github.com/giluis/astray_core/main/blob/src/parsable.rs). 
`Parsable<T>` marks a type as consumable type. This means that given a `TokenIterator<T>`, any struct implementing `Parsable<T>` may be parsed from those tokens.

Its definition:

```rust

pub trait Parsable<T>: std::fmt::Debug
where
    T: Parsable<T>,
    T: ConsumableToken
    Self: Sized,

{

    fn parse(iter: &mut TokenIter<T>) -> Result<Self, ParseError<T>>;

    /* The rest of the trait definition has been omitted. It will be handled later*/
    /* fn parse_if_match(...) */
}
```

Given a `token_iterator: TokenIterator<T>` and `P: Parsable<T>`:

1. `P` shall called a parsable type 
2. `P` may be parsed from token iterator with `P::parse(&mut token_iterator)`
3. `P::parse(&mut token_iterator)` always produces:
    - `Ok(P {/*fields*/})` if parsing succeeds. The iterator is left at the position pointing to the token after the last token that was consumed for parsing `P`
    - `Err(ParseError<T> /*different errors exist*/)`. In this case, the iterator is reset to the position it was before parsing was attempted
2. `T: Parsable<T>` ([with some caveats](./features.md#notes))
    - Calling `<Token as Parsable<Token>>::parse(&mut token_iterator)` just consumes the next token


### TokenIterator 

A `TokenIterator` is a simple collection that goes through a `Vec<Token>`. It can go forwards or backwards.
Each consumable type has a certain length in terms of the tokens it comprises. This length might be [known only at runtime](./features.md#repeated-types).
As an example, considering PseudoRust syntax:
```rust
let a = 1 + 1;
```
The Assignment type has a length of 7 tokens. It could be more, if, for example, the arithmetic expression on the right side of the equal sign had more elements.

`TokenIterator<T>` has 3 important functions:

#### TokenIterator::try_do

```rust
fn try_do<F, Q, E>(&mut self, f: F) -> Result<Q, E>
    where F: FnOnce(&mut TokenIter<T>) -> Result<Q, E>,
```

try_do executes `f`. `f` is will modify the iterator on execution. 
If `f` returns Ok(Q), try_do returns the same and leaves the iterator be.
If `f` returns Err(E), try_do try_do restores the iterator back to the place it was before executing `f`, and returns that Err

#### TokenIterator::parse

```rust
fn parse<P>(&mut self) -> Result<P,ParseError<T>> 
    where P: Parsable<T> {
        Self::try_do(P::parse)
    }
```

As you can see, `parse` is just a wrapper that passes `<P as Parsable<T>>::parse` to `try_do`.
If parsing succeeds, the iterator will stay at where the `parse` function left it
If not, it will reset the iterator to the place it was before the `parse` function was executed.

Why does `TokenIterator<T>::parse` exist, if it's just passing `<Self as Parsable<T>>::parse` to `TokenIterator<T>::try_do`?
Because it allows very concise and clean auto implementation of <Self as Parsable<T>>::parse, [as we'll see in the next chapter](./features.md#structs-enums-and-automatic-derivation).

## structs, enums and automatic derivation

Deriving `SN`, which stands for "Syntax Node" means that the Parsable trait will automatically be implemented.
Check below for what automatic implementations for structs and enums look like

[Any](./features.md#notes) type that derives
```rust
set_token!(Token)

// This parsable type parses 1 token, then 1 Sign, and then another Token
#[derive(SN)]
struct Expression {
    left: Token,
    sign: Sign,
    right: Token, 
}


// This type parses either any token, or any token (useless as is)
#[derive(SN)]
enum Sign {
    Plus(Token),
    Minus(Token),
}

/* generated by Astray */

impl Parsable<Token> for Expression {
    fn parse(iter : &mut TokenIterator) -> Result<Expression, ParseError<T>> {
        let left = iter.parse()?;
        let sign = iter.parse()?;
        let right  = iter.parse()?;
        Ok(Expression{left, sign, right})
    }
}

impl Parsable<Token> for Sign {
    fn parse(iter : &mut TokenIterator) -> Result<Sign, ParseError<T>> {
        let plus = iter.parse().hatch()?;
        let minus = iter.parse().hatch()?;
        // We'll cover errors in the future
        Err(ParseError::disjunct())
    }
}

```

Some key points about the scenario above:
1. Both in `struct`s and `enum`s fields / variants are parsed top to bottom. 
2. Parsable types are composable. Notice how parsing `Tuple` requires parsing `Sign`.
3. [hatch_result](https://crates.io/crates/hatch_result) is used to allow concise enum parsing. It returns on Ok values, rather than Err values. An Err value is constructed in the final line of Sign::parse since reaching that line means no enum variant could be parsed.
4. Both Tuple and Sign parse any Token from the iterator. This is, of course, rather useless in a real scenario, where we would want specific Tokens to be parsed. Parsing specific Tokens requires pattern matching, which we will tackle next. Of course, in this scenario, only the Sign::Plus(_) variant would be parsed, since it is attempted first
5. `iter.parse` is a useful wrapper around <P as Parsable<T>>::parse that keeps the state of the iterator if parsing fails. .
6. Before doing any automatic implementation with Astray, a user must call `set_token!{<UserDefinedToken>}`, to let Astray know for which Tokens our AST will consume. There is an ongoing attempt to remove this necessity.


## Pattern Matching 

Luckily, Astray makes this very easy with the `pat` (stands for "pattern") attribute and the `parse_if_match`
Of course, parsing any token isn't particularly useful. So, we might want to ask for specific variants of our Token type to be consumed for a particular field.
The `Parsable<T>` trait provides a `parse_if_match` function which takes a validation function as argument, representing.

```rust

#[derive(SN)]
struct Expression {
    #[pat(Token::LiteralInt(_))]
    left: Token ,
    sign: Sign,
    #[pat(Token::LiteralInt(_))]
    right: Token, 
}


#[derive(SN)]
enum Sign {
    #[pat(Token::Plus)]
    Plus(Token),
    #[pat(Token::Minus)]
    Minus(Token),
}

/* generated by Astray */

impl Parsable<Token> for Expression {
    fn parse(iter : &mut TokenIterator) -> Result<Expression, ParseError<T>> {
        // Astray uses parse_if_match
        let left = iter.parse_if_match(|tok|matches!(tok, Token::LiteralInt(_)))?;
        let sign = iter.parse()?;
        let right  = iter.parse_if_match(|tok|matches!(tok, Token::LiteralInt(_)))?;
        Ok(Expression{left, sign, right})
    }
}

impl Parsable<Token> for Sign {
    fn parse(iter : &mut TokenIterator) -> Result<Sign, ParseError<T>> {
        let plus = iter.parse_if_match(|tok|matches!(tok, Token::Plus))
                       .hatch()?;
        let minus = iter.parse_if_match(|tok|matches!(tok, Token::Plus))
                        .hatch()?;
        Err(ParseError::disjunct())
    }
}
```

## Further looking into the Parsable<T> trait: Pattern matching

Luckily, the Parsable<T> trait also contains the `parse_if_match` function that will help us with this!
`parse_if_match` will parse a type from the TokenIterator while and apply a validation function to it. 
TokenIterator also has a parse_if_match function that wraps `Parsable<T>::parse_if_match`

```rust
trait Parsable<T>
where 
    T: Parsable<T>,
    Self:Sized,
    T:  ConsumableToken {
        /**
         * The type that should be consumed by the validation function 
         * By default it is Self, but some types namely container types, might want to apply the match function to their contained types, hence why this associated type exists
         * It can be overridden when implementing Parsable<T>. Further examples will be shown of this
         */ 
        type ApplyMatchTo: Parsable<T> = Self;


    /*... fn parse ...*/

    /**
     * default implementation of parse_if_match first parses a type and then applies the validation function to it, return ok only if validation succeeds_
    */
    fn parse_if_match<F: Fn(&Self::ApplyMatchTo) -> bool>(
        iter: &mut TokenIter<TToken>,
        validation: F,
        pattern_str: Option<&'static str>
    ) -> Result<Self, ParseError<TToken>>
    where
        Self: Sized {
            let ty = Self::parse(iter)?;
            if validation(ty) {
                Ok(ty)
            } else {
                Err(ParseError::parsed_but_unmatching(pattern_str))

            }
        }
    



}

```

Luckily, Astray makes this very easy with the `pat` (stands for "pattern") attribute and the `parse_if_match`


```rust

#[derive(SN)]
struct Expression {
    #[pat(Token::LiteralInt(_))]
    left: Token ,
    sign: Sign,
    #[pat(Token::LiteralInt(_))]
    right: Token, 
}


#[derive(SN)]
enum Sign {
    #[pat(Token::Plus)]
    Plus(Token),
    #[pat(Token::Minus)]
    Minus(Token),
}

/* generated by Astray */

impl Parsable<Token> for Expression {
    fn parse(iter : &mut TokenIterator) -> Result<Expression, ParseError<T>> {
        // Astray uses parse_if_match
        let left = iter.parse_if_match(|tok|matches(tok, Token::LiteralInt(_)))?;
        let sign = iter.parse()?;
        let right  = iter.parse_if_match(|tok|matches(tok, Token::LiteralInt(_)))?;
        Ok(Expression{left, sign, right})
    }
}

    fn parse(iter : &mut TokenIterator) -> Result<Sign, ParseError<T>> {
        let plus = iter.parse_if_match(|tok|matches(tok, Token::Plus))
                       .hatch()?;
        let minus = iter.parse_if_match(|tok|matches(tok, Token::Plus))
                        .hatch()?;
        Err(ParseError::disjunct())
    }
}
```
